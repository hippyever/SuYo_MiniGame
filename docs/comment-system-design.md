# 观测回声评论系统设计文档

## 1. 设计目标

“观测回声”是作品详情中的公开交流空间。它帮助试玩者留下具体评价、截图与建议，也让作品成员以“开发者”身份参与讨论。

评论区必须满足三个目标：

1. 作品始终是视觉主体，评论不能抢走登陆行星的叙事重心。
2. 交流方式足够直接，新用户能立即理解评论、回复、点赞和图片上传。
3. 视觉继续属于真实天文观测站，不出现普通社交平台的气泡、热榜和娱乐化动效。

## 2. 设计读取

- 页面类型：现有深空观测站的保留式产品扩展。
- 主要用户：试玩作品的玩家、参赛开发者、活动管理员。
- 视觉语言：理性观测记录、光谱信号、缓慢而克制的宇宙仪式。
- 技术方向：沿用项目现有原生 CSS、CSS 变量和交互组件，不引入另一套设计系统。
- `DESIGN_VARIANCE: 6`：讨论面板保留不对称空间关系，但长内容采用稳定单列。
- `MOTION_INTENSITY: 4`：只为状态变化提供短暂反馈，不做持续干扰文字的动画。
- `VISUAL_DENSITY: 7`：信息密度接近观测记录终端，依靠层级和留白保持可读。

## 3. 需要保留的现有设计资产

- 深空暗色主题与现有浅色降级主题。
- 单一荧光信号色 `--signal`，不为评论区新增第二强调色。
- 现有显示字体、正文字体与等宽数据字体。
- 面板、输入框和按钮保持零圆角。
- 头像可以是圆形天体信标，其余内容容器不使用圆角卡片。
- 作品详情仍是主视图，评论是详情底部的次级入口。
- 桌面使用右侧独立空间，手机使用全屏空间。

## 4. 叙事词汇

| 常规含义 | 页面文案 |
| --- | --- |
| 评论区 | 观测回声 |
| 辅助说明 | 评论与交流 |
| 发表评论 | 发出回声 |
| 回复 | 回应信号 |
| 点赞 | 共鸣 |
| 标签 | 观测标记 |
| 完整评论空间 | 回声频段 |
| 作品成员身份 | 开发者 |
| 作者软删除 | 这段回声已由发出者收回 |
| 管理员隐藏 | 这段信号暂不可见 |
| 空状态 | 尚未接收到观测回声 |

叙事词汇必须带有功能解释。例如标题使用“观测回声”，其下同时出现“评论与交流”，不能只使用隐喻。

## 5. 信息架构

### 5.1 作品详情预览

作品详情底部增加“观测回声”区域：

- 默认展示最新 4 条一级评论，可作为后台配置项调整为 3-5 条。
- 每条只展示作者、时间、开发者标识、正文摘要、最多 3 张图片缩略图、标签、共鸣数和回复数。
- 如果存在回复，只预览最早一条仍可见回复。
- 显示“查看其余 N 条回应”与“进入讨论”。
- 没有评论时显示“尚未接收到观测回声”和“成为第一个留下评论的人”。

### 5.2 完整讨论空间

- 桌面端：从作品详情右侧打开独立讨论面板，作品画面留在背景上下文中。
- 手机端：进入占满可用视口的全屏评论页，使用 `100dvh` 处理浏览器地址栏变化。
- 首次加载 15 条一级评论，可配置为 10-20 条。
- 接近列表底部时自动加载下一批 15 条。
- 一级评论按最新发布时间从新到旧排列。
- 完整回复视图按发布时间从旧到新排列，保证对话可顺序阅读。

### 5.3 两层回复结构

- 第一级是对作品的评论。
- 第二级是该评论下的全部回复。
- 回复可以指向一级评论或另一条回复，但视觉上不继续缩进。
- 指向某条回复时显示“回复 @姓名”。
- 用户可以回复自己，用于补充原评论。
- 默认列表只预览一条回复，其余回复进入该评论的完整讨论视图。

## 6. 评论单元设计

评论单元不是聊天气泡，而是一段被接收的观测记录。

### 6.1 视觉层级

1. 左侧圆形观测者头像，使用稳定天体纹理或用户自定义头像。
2. 姓名作为主要身份，前台不显示邮箱和队伍。
3. 当前属于本作品的成员显示“开发者”，使用信号色文字和细线，不使用高饱和徽章。
4. 发布时间与数据编号使用等宽字体和次级颜色。
5. 正文使用高对比正文字体，保留换行与连续空格，不解析 Markdown、HTML 或富文本。
6. 图片使用清晰矩形网格，点击进入原图查看器。
7. 标签放在正文与操作区之间，采用细边界文本标记，不覆盖图片。
8. 底部操作只保留“回应信号”“共鸣”和回复数量。

### 6.2 图片布局

- 1 张图片：使用单张宽幅缩略图，但限制最大高度。
- 2 张图片：双列等宽。
- 3-5 张图片：三列网格，手机端保持三列缩略图并提供足够点击区域。
- 原图查看器支持左右切换、关闭、图片序号和缩放。
- 图片被管理员移除后，原位置显示“图片已由管理员移除”。
- 图片未加载时使用固定比例骨架，避免页面跳动。

### 6.3 标签维度

官方标签按以下维度组织：

- 玩法体验：玩法有趣、机制新颖、策略丰富、重复游玩性强。
- 视听表达：美术出色、声音出色、氛围沉浸、演出惊喜。
- 操作体验：操作流畅、引导清晰、反馈明确、操作待优化。
- 完成状态：完成度高、仍有潜力、希望继续开发、期待更多内容。
- 建设性建议：平衡性待优化、节奏待调整、性能待优化、发现疑似问题。

标签选择器先显示官方词表，再提供“添加自定义标记”。自定义标签最多 8 个字，仅属于当前评论，不进入公共标签库。

## 7. 发表体验

### 7.1 登录状态

- 未登录用户可以完整阅读。
- 点击“发出回声”或“回应信号”时打开现有邮箱验证流程。
- 验证成功后返回原作品和原评论上下文。
- 当前作品成员在自己的作品下只参与回复，不创建一级评价；在其他作品下按普通已验证用户参与。
- 禁言用户仍可阅读和点赞，但输入区显示禁言截止时间，不允许发表。

### 7.2 编辑器

- 一级评论最多 2,000 字，回复最多 1,000 字。
- 支持普通文字、换行和连续空格。
- 一级评论可选择 0-5 个标签，回复不显示标签选择器。
- 一级评论和回复都能上传 0-5 张图片。
- 文字与图片至少存在一项，标签不能单独发布。
- 不保存本地或云端草稿，关闭空间后未发布内容丢弃。
- 不增加发布前二次确认，按钮只在内容有效且上传完成后可用。

### 7.3 上传反馈

- 每张图片显示独立进度、完成、失败与重试状态。
- 某张失败不影响其他图片。
- 用户可以在发布前移除任意图片。
- 上传过程保持静音。

## 8. 动效与声音

- 发布成功：新记录在当前位置以 180-260ms 的透明度与轻微位移收束，同时出现一次短暂信号扩散。
- 点赞成功：共鸣图标和数字产生一次 140-200ms 的光谱脉冲。
- 打开评论面板：桌面从右侧进入，作品详情保持可见；手机使用整页状态切换。
- 站内通知定位：自动打开目标楼层，并用 1.2 秒以内的边界亮度变化提示目标。
- 粒子只存在于面板边缘和空白区域，不能穿过正文、输入框和图片。
- 评论发布、回复、点赞和通知均保持静音。
- `prefers-reduced-motion` 下取消位移、扩散与脉冲，只保留即时状态变化。

## 9. 通知中心

- 全站顶部工具区增加通知入口和未读数量。
- 通知分为回复、点赞和作品新评论三类偏好。
- 回复逐条通知被回复者。
- 多个点赞按同一评论或回复聚合为“有 5 人共鸣了你的评论”。
- 每类通知可以独立关闭。
- 当前作品成员可以单独开启“本作品新评论”，默认关闭。
- 点击通知进入对应作品和楼层，用户界面不提供复制评论链接。
- 支持“全部已读”。
- 原内容删除或隐藏后，相关通知从用户通知中心消失。

## 10. 状态设计

### 10.1 加载

使用与真实评论布局一致的头像、两行文字和图片占位骨架，不使用居中旋转图标。

### 10.2 空状态

以稀疏的信号波形和明确文案组成，不使用卡通星球：

> 尚未接收到观测回声
> 成为第一个留下评论的人

### 10.3 只读状态

作品撤回、全站暂停或单作品暂停时：

- 保留全部可见内容和图片。
- 隐藏或禁用编辑器。
- 顶部明确说明只读原因。
- 作品重新提交或管理员恢复后，编辑器原位恢复。

### 10.4 删除与隐藏

- 作者删除且无后续回复：前台移除整条内容。
- 作者删除且存在后续回复：保留头像位置与讨论结构，显示“这段回声已由发出者收回”。
- 管理员隐藏：显示“这段信号暂不可见”，回复结构继续保留。
- 管理员查看时显示原内容、当前状态和处理记录。

## 11. 响应式与可访问性

- 所有可点击控件在手机端至少 44px。
- 面板打开后正确管理焦点，关闭时返回触发按钮。
- 图片查看器、回复面板和通知定位支持键盘操作。
- 头像和图片提供可理解的替代文本。
- 评论正文、辅助文字、输入提示和错误提示满足 WCAG AA 对比度。
- 无限加载同时提供屏幕阅读器可感知的加载状态和加载结果数量。
- 触底失败时提供明确的“重新加载”按钮，不能让列表永久停在骨架状态。

## 12. 明确不做

- 不做星级评分、好评率、评论热度排序或作品评论榜。
- 不做无限层级回复、私信、用户举报、开发者删评、置顶和采纳标记。
- 不允许当前作品成员为自己的作品创建一级评价，只允许以开发者身份回应已有评论。
- 不做 Markdown、HTML、富文本、任意用户搜索和群体 @。
- 不做公开评论分享链接。
- 不做邮件通知和未发布草稿同步。
- 不把评论加入作品 ZIP 归档。
- 不使用发布或点赞音效。
